## 1. 專題摘要與主題關聯

**Math Defense** 是一款教育型塔防遊戲(Tower Defense)。它的核心設計理念是:**「數學是遊戲機制本身,而不是進入遊戲的門檻」**。

在多數「數學遊戲」中,數學題只是擋在遊戲樂趣前面的關卡——答對才能繼續玩。我們反其道而行:在 Math Defense 裡,**你所選擇的函數、掃過的弧度、計算的極限,本身就是攻擊**。整個遊戲世界是一個座標平面,玩家要防守的目標 `P*` 是當前關卡中「所有路徑曲線的唯一共同交點」,被隨機放置在平面上的某一點;敵人從格線邊緣沿著程式自動生成的多項式曲線朝 `P*` 前進,玩家則建造對應到不同數學概念的「塔」來阻止它們。

**與「遊戲」主題的關聯性**:本專題是一款可完整遊玩、具備關卡、計分、排行榜與班級對戰的電腦遊戲,題目相關性明確。

本專題的技術亮點在於採用**三層執行環境**:以 Vue 3 + TypeScript 撰寫的前端遊戲引擎、以 FastAPI + PostgreSQL 撰寫的後端服務,以及以 **C 語言**撰寫、編譯為 WebAssembly 的數學運算核心。其中,遊戲的計分公式與關卡生成完全由 C 語言實作,並可在後端**逐位元重新驗算**以防止作弊——這是本專題在技術面的核心特色。

## 3. 遊戲玩法介紹

### 3.1 遊戲流程

一場遊戲(run)的流程如下:

```
主選單 → 選關(★1–5) → 初始作答(Initial Answer) → [ 建造階段 → 進攻波次 ] × N → 結算
                                                       ↑               ↓
                                                  (退費、商店)   蒙提霍爾、連鎖律事件
```

- **初始作答(Initial Answer, IA)**:每關開始前,系統會以分數形式列出本關所有敵人路徑的方程式。這些曲線剛好只有**一個共同交點**,也就是玩家要防守的目標點 `P*`;該點在畫面上以**可自訂的圖示**——星星、猩猩,或玩家自行上傳的客製圖片——加以標示。為了縮小搜尋範圍,系統另會給出一個保證包含此交點的**數值範圍**(`x`、`y` 的上下界)。玩家須親手解這組聯立方程式,算出該交點的精確座標 `(x, y)`,並以分數或整數輸入(例如 `3/2`、`-5/4`)。作答結果有四種:
    - **答對** → `IA = 1`,使計分公式的指數更有利(分數更高),遊戲中正常顯示路徑。
    - **答錯** → `IA = 0`,系統公布正解,路徑仍會顯示。
    - **花 50 金跳過** → `IA = 0`,路徑仍會顯示。
    - **直接開始(隱藏路徑)** → `IA = 0`,且**遊戲中完全不顯示敵人路徑**,須在看不見路徑的情況下作戰,難度最高。
- **建造階段(Build)**:無敵人,可放置 / 升級 / 退費塔、設定每座塔的數學參數(曲線、弧度、向量配對、極限答案、微分 / 積分)、逛商店、設定鎖定目標策略。此階段為「準備時間」,**不計入計分的有效時間**。
- **進攻波次(Wave)**:塔自動開火,玩家仍可施放法術與加速 / 減速;塔的設定在波次中鎖定。

遊戲進行中還可能觸發蒙提霍爾與連鎖律兩種特殊事件,詳見 §3.3。

### 3.2 七種數學塔

每一種塔都對應一個真實的數學主題,玩家操作的數學運算「就是」攻擊:

| 塔 | 符號 | 數學概念 | 運作方式 |
|---|---|---|---|
| Magic | ✦ | 函數圖形(多項式 / 三角 / 對數) | 將圖形畫成帶狀區域,可切換 Debuff(傷害敵人)或 Buff(強化我方塔) |
| Radar A — 掃描 | ◐ | 弧度、扇形面積 | 連續範圍掃描,每 tick 對扇形內所有敵人造成傷害 |
| Radar B — 速射 | ◑ | 弧度 | 受限弧度內的快速單體彈,冷卻最短 |
| Radar C — 狙擊 | ◒ | 弧度 | 緩慢、高傷、長射程的單體重擊 |
| Matrix | ⊞ | 向量與內積 | **成對運作**,基礎傷害 = 配對向量座標的內積;雷射鎖定越久傷害越高 |
| Limit | ∞ | 極限(lim x→a) | 出一道極限選擇題,`+∞` 秒殺範圍內所有敵人、`+C` 依 `|C|×1.5` 放大傷害、其餘答案造成削減傷害 |
| Calculus | ∫ | 微分與積分(冪法則) | 玩家選一多項式並施以 `d/dx` 或 `∫`,依結果生成數隻自律寵物,行為由次數 `n` 決定 |

此外,三種 Radar 塔(A / B / C)都可開啟「**限制攻擊弧度(Restrict attacks to arc)**」選項:預設為關閉(全圓掃描),開啟後玩家可在設定面板中以角度劃出一段扇形範圍 `θ ∈ [θ_start, θ_end]`,塔便**只攻擊落在該弧內的敵人**。弧度越窄、火力越集中;弧度越寬、能守備的路徑越多——玩家可依路徑彎折的位置調整。

每座塔有 Tier 2 / Tier 3 升級(+25% / +50% 傷害、+10% / +20% 射程,加上各塔特化加成)。

### 3.3 敵人與特殊事件

遊戲共有 **10 種敵人**,各有剋制玩法:一般兵、快速兵、強壯兵、分裂兵(死亡分裂)、輔助兵(光環需優先擊殺)、再生兵(需爆發傷害)、堡壘兵(只受 40% 塔傷)、蟲群(只受 35% 塔傷)、Boss Type-A(護盾 + 召喚)、Boss Type-B(觸發連鎖律挑戰)。其中堡壘兵與蟲群的傷害減免**可被寵物與玩家效果繞過**,構成戰術深度。

- **蒙提霍爾事件**:暫停波次,玩家選一扇門,系統揭露一扇必輸的門,可選擇「維持」或「更換」。更換的勝率是 `(門數−1)/門數`(3 門時為 2/3)。遊戲**不會告訴你**這個機率——這正是要傳達的機率課。
- **連鎖律挑戰**:Boss Type-B 半血時出題,答對直接秒殺 Boss + 100 金;答錯不扣血也不懲罰,Boss 回到戰鬥。

### 3.4 計分公式

計分由 C 語言的 `compute_total_score`(`wasm/math_engine.c`)計算,後端可逐位元重算驗證,前端僅鏡像顯示:

```
activeTime = max(0.001, timeTotal − Σ(建造階段花費時間))
S1         = killValue / activeTime                    (擊殺效率)
S2         = killValue / costTotal     (costTotal > 0,否則為 0)  (成本效率)
alpha      = S1 / (S1 + S2)            (S1 + S2 > 0,否則為 0)
K          = alpha·S1 + (1 − alpha)·S2                  (連續混合)
exponent   = 1 / sqrt(max(1, 1 + (2 + healthOrigin − healthFinal − initialAnswer)))
TotalScore = max(0, K)^exponent
```

公式鼓勵玩家「又快又省地擊殺敵人」,並透過 `initialAnswer` 與剩餘血量影響指數,把學習成效(答對開局題、少掉血)反映到分數上。

## 4. 系統架構

### 4.1 三層執行環境

Math Defense 由三個鬆耦合的執行層構成:

```
瀏覽器
  └─ Vue 3 SPA(單頁應用)
       ├─ Pinia 狀態管理(反應式橋接)
       ├─ 遊戲引擎(ECS 風格系統、Canvas 渲染、固定 60 FPS)
       │    ├─ 音效資源管理器
       │    ├─ 錄製 / 重播 / 觀戰
       │    └─ WasmBridge → math_engine.wasm(C 語言, Emscripten)+ JS 後備
       └─ 服務層 → FastAPI 後端
                      ├─ Routers(薄控制器)
                      ├─ Application Services(應用服務)
                      ├─ Domain Aggregates(領域聚合)
                      └─ SQLAlchemy Repositories → PostgreSQL
```

- **前端**負責所有遊戲邏輯、渲染與互動;即使後端離線,單機遊玩仍可運作。
- **C / WASM 數學核心**嵌在前端,處理數值密集的運算(扇形命中判定、積分、矩陣、關卡生成、計分)。
- **後端**負責帳號、排行榜、班級與領土賽、成就 / 天賦、重播驗證等持久化與防作弊。

### 4.2 技術棧

| 層次 | 技術 |
|---|---|
| 前端 | Vue 3.5(Composition API)、TypeScript 6.0(strict)、Pinia 3、Vue Router 5、Vite 8、Vitest 4 |
| 後端 | FastAPI、Uvicorn、SQLAlchemy 2.0、Pydantic v2、PyJWT(HS256)、bcrypt |
| WASM | C99、Emscripten(`-O2`、`-sMODULARIZE -sEXPORT_ES6`、決定論浮點旗標);對外匯出 17 個數學函式 |

| 容器化 | Docker、Docker Compose |
| 重播 | 版本化重播;後端以 `wasmtime-py` 載入同一份 `.wasm` 重算分數 |

### 4.3 共用常數

`shared/game-constants.json` 是前後端共用的單一真實來源(畫布 1280×720、格線範圍、玩家初始 20 HP / 200 金、目標 60 FPS、各星等經濟參數等),避免前後端參數漂移。

## 5. 技術實作重點

### 5.1 C 語言數學引擎(滿足「至少一部份使用 C 語言」)

`wasm/` 目錄下以 **C99** 撰寫,共四個翻譯單元連結為單一 `math_engine.{js,wasm}`:

- `math_engine.c` — 塔的核心運算:2×2 矩陣乘法(Matrix 塔)、扇形面積與**點是否在扇形內**的命中判定(三種 Radar 塔每 tick 對每個敵人呼叫)、梯形法數值積分(Calculus 塔傷害)、以及計分公式 `compute_total_score`。
- `curve.c` — 多項式 / 三角 / 對數三族曲線的求值、微分與定義域判斷(供 Magic 塔與曲線渲染)。
- `level_gen.c` — 關卡生成:交點求解(變號掃描 + 二分逼近)、出生點計算、以及完整的 `generate_level` 拒絕取樣迴圈(8 批 × 50 次嘗試,挑選 `P*`、生成通過該點的 N 條曲線、驗證斜率分離、計算出生點與揭露區域)。
- `prng.c` — PCG XSL-RR 64/32 偽隨機數產生器(供版本 2 重播)。

為何用 C/WASM?核心塔機制每一幀都要跑緊密的數值迴圈(每個敵人、每個 Radar tick 的扇形判定等),編譯後的二進位明顯快於直譯的 JS,並讓數學層與 UI / DOM 解耦。前端永遠透過 `WasmBridge` 呼叫,從不直接 import 產出的 glue。

### 5.2 可重播決定論(技術核心亮點)

`replay_version = 2` 要求**跨任何 WASM 引擎都產生位元相同的浮點輸出**,我們透過三點同時成立來保證:

1. Emscripten 把 musl libc 編進 `.wasm`,`sinf` / `cosf` / `logf` / `pow` 等以 WASM bytecode 形式隨二進位附帶,而非主機引擎的內建函式;
2. WASM runtime 逐位元解讀該 bytecode,不替換成自己的超越函數;
3. 編譯旗標 `-fno-fast-math`、`-fno-unsafe-math-optimizations`、`-ffp-contract=off` 禁止編譯器對運算式做會造成 ULP 漂移的代數重寫(例如把 `a*b+c` 融合成 FMA)。

因此,整場遊戲可由 `rng_seed` 完整重現;後端以 `wasmtime-py` 載入**同一份 `.wasm`** 重算分數,凡提交分數與伺服器重算差異超過 `1e-4`,即以 HTTP 422 `replay_mismatch` 拒絕——形成**伺服器端防作弊**機制。`shared/score_parity_fixtures.json` 釘住 C / Python / TS 三方實作的一致輸出,任何公式變動都會被同位性測試攔下。

### 5.3 JS 後備與雙實作同位性

每一個 WASM 函式在 `WasmBridge.ts` 都有純 TypeScript 後備實作。`initWasm()` 在啟動時 await 一次,若 WASM 載入失敗(網路錯誤、瀏覽器不支援、未建置),橋接層會**自動透通**地改走後備路徑,遊戲照常可玩。測試會強制走後備路徑,驗證 WASM 與 JS 在邊界案例輸出一致。

### 5.4 後端領域驅動設計(DDD)分層

後端採 DDD 分層:**Routers(薄控制器)→ Application Services(應用服務)→ Domain Aggregates(領域聚合)→ Repositories(SQLAlchemy)**。領域層不含任何 HTTP 概念,領域錯誤透過全域例外處理器映射為 HTTP 狀態碼。這讓核心業務規則(使用者、遊戲場次、排行榜、成就、天賦、班級、領土、賽季、挑戰、貝氏能力估計)可獨立測試與演進。

### 5.5 前端遊戲引擎

前端採 **ECS(Entity-Component-System)風格**系統、固定 **60 FPS** 主迴圈、Canvas 直接渲染,並具備錄製 / 重播 / 即時觀戰。相位轉換由前端 `PhaseStateMachine` 強制,並由後端 `GameSession` 聚合的狀態機鏡像,前後端對「現在處於哪個階段」有一致認知。

### 5.6 班級與領土競賽

學生加入班級,在限時的「搶領土」活動中以地區 / 班級 / 全球排行榜競爭;每個活動至多 50 格,並有教師可設定的單一學生佔格上限。領土結算由背景 scheduler 處理,即時觀戰以行程內 WebSocket fan-out 實作。

### 5.7 測試與品質

- **後端**:約 32 個測試檔(DDD 聚合、路由、領域不變式、登入鎖定、Token 黑名單、CSRF、計分同位性、重播重算、wasmtime 執行環境等),以真實 PostgreSQL 測試庫執行,行覆蓋率約 **81%**。
- **前端**:約 87 個測試檔(系統、引擎、WASM / JS 同位性、重播決定論、鍵盤放置、計分同位性等),以 Vitest + happy-dom 執行。
- **壓力測試**:`stress/` 下另有 k6 與計算基準測試。

## 6. 創意、新意與原創性

- **數學即機制**:每座塔對應一個真實數學主題,且該主題的運算就是攻擊本身(內積決定 Matrix 傷害、積分決定 Calculus 傷害、極限答案決定 Limit 效果),而非「答對才能玩」。這是與一般數學練習 App 最根本的差異。
- **把機率課藏進遊戲**:蒙提霍爾事件刻意不揭露「換門較有利」的數學事實,讓玩家自行從重複遊玩中體會——這是設計層面的教學創新。
- **以座標平面為戰場**:防守目標 `P*` 是「所有曲線的唯一共同交點」,敵人沿程式生成的多項式曲線行進,將「函數圖形」與「交點」這些抽象概念視覺化、遊戲化。
- **可驗證的決定論**:用 C/WASM 達成跨引擎位元一致,讓伺服器能重算分數防作弊——兼具技術性與工程嚴謹度。
- **完整的教育閉環**:成就、天賦樹、班級領土賽與排行榜構成長期進程,並以貝氏能力估計追蹤學習狀態。

本專題的設計理論依據另見 `docs/Educational_Theory_Analysis.md`。

## 8. 安裝與執行方式(Ubuntu 24.04)

完整步驟見專案 `README.md` 與 `UBUNTU_SETUP.md`。以下為摘要,確保助教能順利執行。

### 8.1 環境需求

- Node.js 26+、Python 3.13+、PostgreSQL 16
- Docker 與 Docker Compose(建議路徑)
- Emscripten SDK(僅在需要重新編譯 WASM 時;`emsdk/` 已隨附,且編譯產物已 commit,故一般無需重建)

### 8.2 方式 A — Docker(建議)

```bash
cp .env.example .env     # 編輯 .env:設定 SECRET_KEY(≥32 字元)、
                         # 替換 DATABASE_URL 與 POSTGRES_PASSWORD 中的 changeme、
                         # 設定 TOTP_ENCRYPTION_KEY(後端未設妥不會啟動)
docker compose up
```

- 前端:http://localhost:5173
- 後端 API:http://localhost:8000　健康檢查:http://localhost:8000/health

### 8.3 方式 B — 手動

需先有運作中的 PostgreSQL 16 及 `math_defense` 資料庫 / 角色(指令見 `backend/README.md`)。

```bash
# 後端
cd backend && python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp ../.env.example .env     # 將 DATABASE_URL 主機由 postgres 改為 localhost
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

# 前端(另開終端機)
cd frontend && npm install && npm run dev
```

資料庫遷移會在後端首次啟動時透過 FastAPI lifespan 自動套用。

### 8.4 重建 WASM(選用)

```bash
cd wasm && make    # 需要 PATH 上有 emcc;產物寫入 frontend/src/math/wasm/
```

僅在修改 `wasm/*.c` 時需要;`npm run build` 會透過 `prebuild` 自動觸發。

## 9. 開發歷程與遭遇的困難

- **時程**:依課程節點推進——05/12 proposal、06/09 上台報告、06/12 繳交。開發歷經 V1 → V2 → V3 數次重構(git 紀錄可佐證),V2 將早期較雜的塔種整併為 7 種以數學概念為核心的塔。
- **困難一:跨引擎浮點決定論**。最初前端顯示分數與後端重算結果會出現微小差異,根因是編譯器對浮點運算式的重寫與 FMA 融合;解法見 §5.2(浮點旗標 + musl-in-WASM,並以同位性 fixtures 鎖住)。
- **困難二:WASM 記憶體管理**。傳陣列給 C 需手動 `malloc` / `free`。我們在 `WasmBridge.ts` 以 RAII 風格的 `withFloatBuffers` 包裝,確保即使提早 `throw` 也不會洩漏。
- **品質保證**:所有 LLM 輔助產出的程式碼皆經人工審閱、自動化測試(規模見 §5.7)與實機 demo 驗證,以因應課程對「LLM 出錯加重扣分」的條款。

## 11. 結論與未來展望

Math Defense 以「數學即遊戲機制」為核心,證明了數學概念可以是遊戲樂趣的來源而非阻礙。在技術上,我們以三層架構(Vue / FastAPI / C-WASM)落實了一個可遊玩、可競賽、且具備伺服器端防作弊與可重播決定論的完整系統,並兼顧無障礙設計。

**未來展望**:擴充更多數學主題對應的塔(如機率分佈、線性代數)、加入多人即時對戰、以貝氏能力估計做個人化出題與難度自適應,以及將 C 核心進一步抽出為可在原生(host-clang)環境驗證的同位性測試。
