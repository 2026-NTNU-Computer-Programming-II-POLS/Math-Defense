# WASM Math Engine

High-performance math module for Math Defense, written in C99 and compiled to WebAssembly via Emscripten. It implements the numerically intensive operations that back the game's tower mechanics (matrix transforms, sector hit-tests, trapezoid integration, curve evaluation), plus the bit-deterministic replay-v2 primitives (PCG PRNG, level generation, intersection/spawn solvers) and the canonical score formula. Every function also has a pure-TypeScript fallback in `WasmBridge.ts`, so the game remains fully playable even if the WASM module fails to load.

---

## Why WASM?

The core tower mechanics drive tight numerical loops every frame — point-in-sector checks for every enemy per Radar system tick, sector coverage area calculations, matrix transformations, and integration for Calculus tower damage. A compiled binary runs these noticeably faster than interpreted JS and keeps the math layer decoupled from both the UI and the DOM.

The frontend always calls the bridge (`WasmBridge.*`); it never imports `math_engine.js` directly. `WasmBridge.test.ts` asserts that the JS fallback and WASM output agree on boundary cases.

## Files

```
wasm/
├── math_engine.c    Tower mechanics — matrix, sector, integrate, score (C99)
├── prng.c / prng.h  PCG XSL-RR 64/32 PRNG used by replay v2
├── curve.c / curve.h
│                    Curve evaluator/derivative/in-domain for the 3 families
├── level_gen.c / level_gen.h
│                    Level generation: intersection solver, spawn calculator,
│                    full generate_level rejection-sampling loop
├── Makefile         Emscripten build configuration
└── README.md        This file
```

The four translation units link into a single `math_engine.{js,wasm}` pair —
one download, one ABI surface, shared RAII helper on the bridge side.

The build writes directly into the frontend source tree (so Vite bundles the
glue and hashes the `.wasm` as an asset — rather than treating them as opaque
`public/` files, which newer Vite versions refuse to dynamic-import):

```
frontend/src/math/
├── wasm-exports.d.ts          Union of exported symbol names (generated by
│                              the Makefile's `dts` target from EXPORTED_FUNCTIONS)
└── wasm/
    ├── math_engine.js         ES6 module wrapper (generated by Emscripten)
    ├── math_engine.wasm       Binary module (generated by Emscripten)
    └── math_engine.d.ts       Hand-written loose ambient declaration for the
                               glue (Emscripten emits no `.d.ts`; this only
                               lets TS accept the `createMathEngine` import —
                               the real ABI is re-asserted in `WasmBridge.ts`).
                               Not regenerated or removed by `make` / `make clean`.
```

`wasm-exports.d.ts` lives one directory up from the binary because it is
imported by `WasmBridge.ts` as a plain `WasmExport` string-literal union and
should not be co-bundled with the WASM asset. The Makefile regenerates it
whenever the `EXPORTED_FUNCTIONS` list changes, so a rename in the C export
list is caught by `vue-tsc` instead of crashing at runtime.

## Build

### Prerequisites

Install [Emscripten SDK](https://emscripten.org/docs/getting_started/downloads.html) and activate it. The repo vendors `emsdk/` at the project root so you can:

```bash
# From project root
source emsdk/emsdk_env.sh     # Linux/macOS
emsdk\emsdk_env.bat           # Windows
```

> **The vendored `emsdk/` is a Windows build.** The `emsdk_env.sh` line above
> only yields a working `emcc` if `emsdk/` was provisioned for your platform. On
> macOS/Linux, install a native SDK instead (`git clone emscripten-core/emsdk`,
> `./emsdk install 5.0.7 && ./emsdk activate 5.0.7`), or skip host setup entirely
> and let Docker build the binary (`docker compose build backend`). See
> [../UBUNTU_SETUP.md](../UBUNTU_SETUP.md#about-the-wasm-engine).

### Compile

```bash
cd wasm
make
```

The `Makefile` anchors paths to its own directory, so `make -f path/to/Makefile` works from anywhere. The frontend's `npm run build` script triggers `make` automatically via the `prebuild` hook (`"prebuild": "cd ../wasm && make"` in `frontend/package.json`).

The `Makefile` pins the full export list, the runtime helpers, the module
shape, and a determinism-critical FP flag set:

| Flag | Purpose |
|---|---|
| `-sEXPORTED_FUNCTIONS=…` | The 17 math symbols plus `_malloc`/`_free` (the single source of truth — see below) |
| `-sEXPORTED_RUNTIME_METHODS=ccall,cwrap,getValue,setValue` | Runtime helpers the bridge needs: `ccall`/`cwrap` to call exports, `getValue`/`setValue` to read/write the heap for struct- and buffer-passing calls |
| `-sMODULARIZE=1 -sEXPORT_ES6=1` | Emit an ES6 module so the bridge can static-import it through Vite |
| `-sEXPORT_NAME=createMathEngine` | Factory function name used by `WasmBridge.ts` |
| `-sALLOW_MEMORY_GROWTH=1` | Linear memory can grow at runtime |
| `-sMAXIMUM_MEMORY=268435456` | Cap linear memory at 256 MiB — prevents a runaway allocation from starving the rest of the tab |
| `-fno-fast-math` | Forbid algebraic rewrites of transcendentals — required for cross-engine bit-determinism |
| `-fno-unsafe-math-optimizations` | Same — disable reciprocal substitutions |
| `-ffp-contract=off` | Disable FMA fusion. The compiler must not turn `a*b+c` into a fused multiply-add: that would yield engine-dependent ULP drift |
| `-O2` | Optimisation (still safe under the FP flags above) |

### Determinism rationale

`replay_version=2` (construction plan §3.5) requires **bit-identical**
floating-point output across every WASM engine that runs the binary. Three
properties hold simultaneously:

1. emcc compiles musl libc into the `.wasm`, so `sinf`, `cosf`, `logf`,
   `asinf`, `acosf` ship as WASM bytecode — not as host-engine intrinsics.
2. WASM runtimes interpret that bytecode byte-for-byte (no SIMD relaxation
   we use) and never substitute their own transcendentals.
3. The FP flags above prevent the C compiler from rewriting expressions in
   ways that would diverge between toolchain versions.

The parity tests under `frontend/src/math/*.wasm.test.ts` pin this contract.
A future host-clang `parity_test.c` (construction plan §5.3, follow-up)
will extend the guarantee to a non-WASM build of the same TUs.

### Clean

```bash
make clean
```

### Regenerate score parity fixtures

`shared/score_parity_fixtures.json` pins the agreed-on output of `compute_total_score` across the C/Python/TS implementations (see `compute_total_score` below). Whenever the formula changes — even by one ULP — the fixtures must be refreshed so the parity tests keep their teeth:

```bash
cd wasm
make regenerate-fixtures
```

This runs `scripts/regenerate_score_fixtures.py`, which loads the Python mirror (`backend/app/domain/scoring/score_calculator.py`) and re-evaluates the canonical input matrix defined in the script. The input matrix is the source of truth for "what cases we care about"; add a new entry there when you introduce a new branch in the formula. The required order of operations during a formula change is documented at the top of that script.

## Exported Functions

All functions are exposed as C symbols prefixed with `_` (Emscripten convention) and pinned by the Makefile's `EXPORTED_FUNCTIONS` list (single source of truth). `WasmBridge.ts` statically imports `createMathEngine` from the generated `math_engine.js` glue and wraps individual exports via `cwrap()` / direct `Module._<name>` calls. The matching `WasmExport` string-literal union in `frontend/src/math/wasm-exports.d.ts` is regenerated from the same list so the bridge cannot reference a name that does not exist.

> **Heap helpers**: `_malloc` and `_free` are also exported (the bridge calls them as `Module._malloc` / `Module._free` for buffer-passing calls) but are omitted from the `WasmExport` union because they are not accessed through `ccall` / `cwrap`.

### `matrix_multiply`

```c
void matrix_multiply(const float *a, const float *b, float *result);
```

Standard 2×2 matrix multiplication. `a`, `b`, `result` are flat 4-float arrays `[a00, a01, a10, a11]`. Used by the **Matrix** tower to compose rotation/scaling transformations.

---

### `sector_coverage`

```c
float sector_coverage(float radius, float angle_width);
```

Area of a circular sector: `0.5 · r² · θ`. Used by the **Radar A** tower for damage scaling by coverage.

---

### `point_in_sector`

```c
int point_in_sector(
    float px, float py,
    float cx, float cy,
    float radius,
    float angle_start,     // radians
    float angle_width      // radians
);
// returns: 1 inside, 0 outside
```

Point-in-sector hit test. Core per-tick check for the **Radar A**, **Radar B**, and **Radar C** towers.

---

### `numerical_integrate`

```c
float numerical_integrate(
    float coeff_a, float coeff_b, float coeff_c,   // y = ax² + bx + c
    float lo, float hi,
    int n                                           // trapezoid subdivisions
);
```

Trapezoid-rule approximation of `∫[lo,hi] |a·x² + b·x + c| dx` with `n` subdivisions (defaulted to 100 if non-positive). Each sample's `|y|` is taken before the sum, so the result is the area between the curve and the x-axis rather than the signed integral. Used by the **Calculus** tower to compute damage from the player's chosen function.

---

### `compute_total_score`

```c
double compute_total_score(
    double kill_value,
    double time_total,
    double prep_sum,
    double cost_total,
    double health_origin,
    double health_final,
    int    initial_answer        /* 0 or 1 */
);
```

**Canonical V2 score formula** — single source of truth for both the
frontend display (`score-calculator.ts`) and the backend anti-cheat
verifier (`score_calculator.py` via `app/infrastructure/wasm_runtime.py`,
which mounts this same `.wasm`). The s1/s2/k/exponent breakdown shown in
`ScoreResultView` is derived locally, but `totalScore` always comes out
of this function so server and client agree to the last ULP.

`prep_sum` is pre-summed by the caller so the ABI stays a flat scalar
list and the parity fixtures in `shared/score_parity_fixtures.json` don't
need to encode variable-length prep arrays. A v2 session whose submitted
`total_score` diverges from the server recompute by more than `1e-4` is
rejected with HTTP 422 + `replay_mismatch`. Any change to this function
MUST be mirrored in both fallbacks (TS / Python) and the parity fixtures
regenerated.

### `power_f64`

```c
double power_f64(double base, double exp);
```

Bit-deterministic `pow` via musl's `pow` compiled into WASM bytecode.
Exposed for callers that need a determinism-guaranteed `pow` outside the
fixed `compute_total_score` shape — and used internally by it.

The cross-engine determinism guarantee comes from §`Determinism rationale`
above: `pow` is musl in WASM bytecode, byte-identical regardless of host.

---

### `prng_seed` / `prng_next_u32` / `prng_next_f64`

```c
typedef struct { uint64_t state; uint64_t inc; } prng_t;   /* 16 bytes */

void     prng_seed     (prng_t *r, uint32_t seed, uint32_t stream);   /* see ABI note */
uint32_t prng_next_u32 (prng_t *r);
double   prng_next_f64 (prng_t *r);   /* [0, 1), 53-bit mantissa */
```

PCG XSL-RR 64/32 (O'Neill, public domain). Pure integer arithmetic — same
cross-engine determinism guarantee as `mulberry32`, but with 128-bit state
and a stream-id channel (`inc`) for splitting `level_rng` from
`gameplay_rng`. `prng_next_f64` builds a `[0, 1)` double from two `u32`
draws as `(top26 * 2^27 + bot27) / 2^53`; rounding never enters because each
intermediate is exactly representable. Used by `Game.setSeed` when WASM is
loaded.

> **ABI note** — the public `prng_seed` signature exposes `seed` and `stream`
> as `uint32_t` rather than the PCG paper's `uint64_t`. The narrowing is
> deliberate: Emscripten's `ccall` marshalling for `i64` arguments forces
> JavaScript `BigInt` round-trips at every call site, which the bridge avoids
> by widening the 32-bit seed internally inside `prng.c`. See the comment in
> `wasm/prng.c` for the rationale.

### `curve_evaluate` / `curve_derivative` / `curve_in_domain`

```c
typedef struct {
    uint32_t family;     /* 0=poly, 1=trig, 2=log */
    uint32_t variant;    /* poly: degree (1|2|3); trig: 0=sin, 1=cos; log: 0 */
    float    a, b, c, d;
} curve_t;               /* 24 bytes */

float   curve_evaluate   (const curve_t *c, float x);
float   curve_derivative (const curve_t *c, float x);
int32_t curve_in_domain  (const curve_t *c, float x);  /* 1 in / 0 out */
```

Polynomial / trigonometric / logarithmic evaluator. Replaces the JS
`curve-evaluator.ts` implementation, which now delegates to the bridge.
Trigonometric variants call musl `sinf`/`cosf` directly; the log variant
calls `logf`. Domain check is `arg > 0` for log, unconstrained otherwise.

---

### `find_pair_intersections`

```c
int32_t find_pair_intersections(
    const curve_t *c1, const curve_t *c2,
    float xmin, float xmax, float step,
    float *out_xs, int32_t max_out
);   /* returns count written */
```

Sign-change scan over `[xmin, xmax]` at granularity `step`, then bisection
to refine each root. Used by the level generator to verify multi-curve
common points. Iteration cap of the bisector is a function of `step / EPS`
computed via `logf` — kept out of JS so the cap itself is engine-stable.

---

### `find_all_curves_common_point`

```c
int32_t find_all_curves_common_point(
    const curve_t *curves, int32_t n,
    float xmin, float xmax, float step,
    float *out_xs, float *out_ys, int32_t max_out
);
```

Returns every `x` in `[xmin, xmax]` where **all** N curves agree on `y`
within `COMMON_PT_TOL = 1e-4`. Hot path during rejection sampling.

---

### `count_common_intersections_in_interval`

```c
int32_t count_common_intersections_in_interval(
    const curve_t *curves, int32_t n,
    float xmin, float xmax
);
```

Cardinality of common intersections — saves the O(N) coordinate marshal
when the caller only needs the count.

### `compute_spawn_points`

```c
typedef struct {
    float    x, y;
    uint32_t edge;        /* 0=top, 1=bottom, 2=left, 3=right */
    int32_t  curve_index;
    int32_t  side;        /* +1 right of P*, -1 left of P* */
} spawn_t;                /* 20 bytes */

int32_t compute_spawn_points(
    const curve_t *curves, int32_t n,
    float endpoint_x, float endpoint_y,
    spawn_t *out_spawns, int32_t max_out
);
```

For each curve, march left and right from the disclosure endpoint; bisect
the boundary crossing. Each curve contributes up to 2 spawns. Output is
ordered (curve_index ascending, then side: +1 then -1).

---

### `generate_level`

```c
typedef struct {
    int32_t  success;        /* 1 ok, 0 exhausted */
    int32_t  curve_count;
    curve_t  curves[8];
    float    endpoint_x, endpoint_y;
    float    region_x_min, region_x_max, region_y_min, region_y_max;
    float    interval_lo, interval_hi;
    int32_t  spawn_count;
    spawn_t  spawns[16];
} generated_level_t;          /* 556 bytes */

int32_t generate_level(
    int32_t              star_rating,        /* metadata only; multiset choice happens TS-side */
    prng_t              *rng_state,
    const int32_t       *multiset_entries,   /* polynomial degrees: 1, 2, or 3 */
    int32_t              multiset_len,
    generated_level_t   *out
);
```

The full rejection-sampling loop: 8 batches × 50 attempts. Each attempt
picks `P*`, generates `N` curves through it, verifies slope separation,
computes spawn points, finds a disclosure region, and checks that the
common point lives inside the y-band. On success, writes the entire
`generated_level_t` and returns 1. On exhaustion, returns 0 — TS surfaces
the failure rather than silently falling back to a v1 level (see
`LevelSelectView.startLevel` for why mixing v1 and v2 layouts breaks
replay).

> **Polynomial-only**: as of the V3 fraction fill-in rework, the level
> generator emits polynomials exclusively. Trig and log multiset wire codes
> (the historical 100 / 101 / 200) were removed from `level_gen.h`. The
> trig/log curve families still exist in `curve.h` for the Magic tower's
> function-curve overlay — the generator just never selects them.

## Memory Management

WASM uses a linear memory heap; array arguments require explicit allocation. `WasmBridge.ts` wraps every buffer-passing call in a RAII helper:

```typescript
// Simplified pattern from WasmBridge.ts (module-private helper)
function withFloatBuffers<T>(
    sizes: number[],
    cb: (...ptrs: number[]) => T,
): T {
    const ptrs = sizes.map(n => wasm._malloc(n * 4))   // 4 bytes / float32
    try {
        return cb(...ptrs)
    } finally {
        ptrs.forEach(p => wasm._free(p))               // always freed
    }
}
```

Callers never see raw pointers; leaks from early `throw` are impossible.

## JavaScript Fallback

Every WASM function has a pure-TypeScript equivalent in `WasmBridge.ts`. `initWasm()` is awaited once at app startup; on failure (network error, unsupported browser, build not run) the bridge transparently routes every subsequent call to the fallback. Game systems always import from `WasmBridge` — they are oblivious to which backend is active.

`isUsingWasm()` is exposed for diagnostics. `setUseWasm(false)` forces the fallback path (used by `WasmBridge.test.ts` to verify parity).

## Tower ↔ Function Mapping

| Tower | WASM Function(s) |
|---|---|
| Magic | `curve_evaluate`, `curve_derivative`, `curve_in_domain` (function-curve overlay; family-aware) |
| Radar A (Sweep) | `point_in_sector`, `sector_coverage` |
| Radar B (Single-target) | `point_in_sector` |
| Radar C (Slow powerful) | `point_in_sector` |
| Matrix | `matrix_multiply` |
| Limit | — (pure JS — limit question generation and outcome resolution) |
| Calculus | `numerical_integrate` |

The level generator (`generate_level`, `find_pair_intersections`,
`find_all_curves_common_point`, `count_common_intersections_in_interval`,
`compute_spawn_points`) is shared infrastructure — not bound to a single
tower. Score recomputation (`compute_total_score`, with `power_f64` as
the underlying determinism-pinned primitive) backs the FU-A backend
recompute path; replay-v2 PRNG (`prng_seed` / `prng_next_u32` /
`prng_next_f64`) is the gameplay random stream.

> **Historical note**: Earlier V1 versions included Function Cannon, Integral Cannon, Fourier Shield, and Probability Shrine, which carried four extra C exports (`calculate_trajectory`, `fourier_composite`, `fourier_match`, `line_circle_intersect`). The V2 redesign (Phase 5, April 2026) unified them into 7 concept-based towers, and those exports were dropped from the binary in May 2026 once V2 was confirmed not to need them.
