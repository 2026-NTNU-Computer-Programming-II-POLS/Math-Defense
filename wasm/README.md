# WASM Math Engine

High-performance math module for Math Defense, written in C and compiled to WebAssembly via Emscripten. It handles all computationally intensive math operations required by the game's tower mechanics.

---

## Why WASM?

The game's tower mechanics (trajectory intersection, sector coverage, numerical integration, Fourier wave matching) involve tight numerical loops that are noticeably faster in a compiled binary than in interpreted JavaScript. Using WASM also keeps math logic decoupled from the UI layer and independently testable.

The frontend includes a pure-JavaScript fallback for every exported function, so the game remains fully playable even if the WASM module fails to load.

## Build

### Prerequisites

Install [Emscripten SDK](https://emscripten.org/docs/getting_started/downloads.html) and activate it:

```bash
# From project root — emsdk is already included
source emsdk/emsdk_env.sh      # Linux/macOS
emsdk\emsdk_env.bat            # Windows
```

### Compile

```bash
cd wasm
make
```

This runs:

```bash
emcc math_engine.c \
  -O2 \
  -o ../frontend/public/wasm/math_engine.js \
  -s MODULARIZE=1 \
  -s EXPORT_ES6=1 \
  -s EXPORTED_FUNCTIONS='[...]' \
  -s EXPORTED_RUNTIME_METHODS='["cwrap","getValue","setValue","_malloc","_free"]' \
  -s ALLOW_MEMORY_GROWTH=1
```

### Clean

```bash
make clean
```

## Exported Functions

All functions are exported as C symbols prefixed with `_` (Emscripten convention). The frontend accesses them through `cwrap()` in `WasmBridge.ts`.

### `matrix_multiply`

```c
void matrix_multiply(
    float* a,   // input: 4-element flat array [a00,a01,a10,a11]
    float* b,   // input: 4-element flat array
    float* out  // output: 4-element result
);
```

Standard 2×2 matrix multiplication. Used by the **Matrix Link** tower to apply rotation/scaling transformations.

---

### `calculate_trajectory`

```c
int calculate_trajectory(
    float a, float b, float c,  // quadratic coefficients y = ax² + bx + c
    float x_start, float x_end,
    float step,
    float* out_x,               // output: x values
    float* out_y                // output: y values
);
// returns: number of points written
```

Evaluates a quadratic polynomial over `[x_start, x_end]` in increments of `step`. Used by the **Function Cannon** to draw and intersect the projectile trajectory.

---

### `sector_coverage`

```c
float sector_coverage(float radius, float angle_width);
```

Returns the area of a circular sector: `0.5 × r² × θ`. Used by the **Radar Sweep** tower for damage scaling.

---

### `point_in_sector`

```c
int point_in_sector(
    float px, float py,         // test point
    float cx, float cy,         // sector center
    float radius,
    float angle_start,          // radians
    float angle_width           // radians
);
// returns: 1 if inside, 0 if outside
```

Tests whether a point falls within a circular sector. Core hit-detection for the **Radar Sweep** tower.

---

### `numerical_integrate`

```c
float numerical_integrate(
    float coeff_a, float coeff_b, float coeff_c,  // y = ax² + bx + c
    float lo, float hi,                            // integration bounds
    int n                                          // trapezoid subdivisions
);
```

Approximates `∫[lo,hi] (ax² + bx + c) dx` using the trapezoid rule with `n` subdivisions. The result is the damage dealt by the **Integral Cannon**.

---

### `fourier_composite`

```c
float fourier_composite(
    float t,            // time parameter
    float* freqs,       // array of 3 frequencies
    float* amps         // array of 3 amplitudes
);
```

Evaluates `Σ aᵢ · sin(2π · fᵢ · t)` for three sine components. Used to generate and render the target wave in the **Fourier Shield** mini-game.

---

### `fourier_match`

```c
float fourier_match(
    float* freqs1, float* amps1,    // target wave (3 components each)
    float* freqs2, float* amps2,    // player input wave
    int samples                      // number of sample points
);
// returns: similarity score in [0, 1]
```

Computes the normalised similarity between two 3-component Fourier waves by sampling both at `samples` points and comparing. Returns `1.0` for a perfect match, `0.0` for completely dissimilar. Used to determine boss shield break in the **Fourier Shield** tower.

---

### `line_circle_intersect`

```c
int line_circle_intersect(
    float m, float b,           // line: y = mx + b
    float cx, float cy,         // circle center
    float r,                    // circle radius
    float* out_x, float* out_y  // output: up to 2 intersection points
);
// returns: number of intersections (0, 1, or 2)
```

Finds intersection points between a line and a circle. Used by the **Function Cannon** to compute where a quadratic trajectory hits an enemy's collision circle.

## Memory Management

WASM uses a linear memory heap. Passing arrays requires explicit allocation:

```typescript
// Pattern used in WasmBridge.ts
const ptr = wasm._malloc(length * 4);   // 4 bytes per float32
try {
    // write input
    for (let i = 0; i < length; i++) {
        wasm.setValue(ptr + i * 4, values[i], 'float');
    }
    wasm._exported_fn(ptr, ...);
    // read output
} finally {
    wasm._free(ptr);                     // always freed
}
```

`WasmBridge.ts` uses `try/finally` around every WASM call to guarantee `_free()` is called even if an exception is thrown.

## JavaScript Fallback

`frontend/src/math/WasmBridge.ts` exposes a unified `MathAPI` interface. If the WASM module fails to load (network error, unsupported browser), every function transparently falls back to a pure-TypeScript implementation:

```typescript
// WasmBridge.ts (simplified)
export const MathAPI = {
    pointInSector: wasmLoaded
        ? (px, py, cx, cy, r, aStart, aWidth) => wasm._point_in_sector(...)
        : (px, py, cx, cy, r, aStart, aWidth) => jsFallback.pointInSector(...),
    // ...
};
```

The game systems always call `MathAPI.*` — they never import from `math_engine.js` directly.
